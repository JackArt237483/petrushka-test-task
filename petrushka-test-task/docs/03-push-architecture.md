# 03. Push Architecture — верхнеуровневая архитектура PUSH-уведомлений

## 1. Цель

В рамках задания требуется предложить верхнеуровневую архитектуру отправки PUSH-уведомлений для мобильного приложения интернет-магазина «Петрушка Зеленая».

Согласно условию, PUSH-уведомления могут быть разных типов:
- уведомления о корзине, которая долго остается без действий;
- уведомления об отмене заказа;
- рекламные рассылки;
- другие типы уведомлений в будущем.

Дополнительно указано, что backend построен на микросервисной архитектуре.  
Следовательно, решение должно быть масштабируемым, расширяемым и не зависеть от одного конкретного сценария.

---

## 2. Основная идея решения

Отправка PUSH-уведомлений не должна быть жестко встроена в отдельный сервис или бизнес-сценарий.  
Правильнее выделить отдельный контур уведомлений, который:

- принимает события от разных микросервисов;
- определяет, нужно ли отправлять PUSH;
- формирует сообщение;
- передает его внешнему push-провайдеру;
- получает статус доставки;
- сохраняет историю отправок.

Иными словами, PUSH-уведомления должны работать как отдельная подсистема, а не как случайная логика внутри Cart Service или Order Service.

---

## 3. Верхнеуровневые компоненты

Ниже приведены основные компоненты архитектуры.

### 3.1. Mobile Application

Мобильное приложение:
- регистрирует устройство пользователя;
- получает device token от push-провайдера;
- передает token на backend;
- принимает PUSH-уведомления;
- открывает нужный экран приложения по нажатию на уведомление.

---

### 3.2. API Gateway / BFF

Слой API Gateway или BFF:
- принимает запросы от мобильного приложения;
- может использоваться для регистрации device token;
- передает запросы во внутренние сервисы.

Этот слой не обязан отправлять PUSH самостоятельно, но может участвовать в управлении настройками уведомлений пользователя.

---

### 3.3. User Service

User Service хранит данные пользователя, необходимые для отправки уведомлений:
- идентификатор пользователя;
- статус пользователя;
- предпочтения по уведомлениям;
- согласие на рекламные PUSH;
- связь между пользователем и устройствами.

---

### 3.4. Device Token Service / Notification Preferences Service

Отдельный сервис или часть notification-контура, отвечающая за:
- хранение device token;
- привязку token к пользователю;
- хранение платформы устройства (`iOS`, `Android`);
- учет нескольких устройств одного пользователя;
- хранение пользовательских настроек уведомлений.

Например:
- разрешены ли рекламные уведомления;
- разрешены ли транзакционные уведомления;
- включены ли уведомления по заказам.

---

### 3.5. Domain Microservices

Источниками событий могут быть разные доменные сервисы:

- **Cart Service** — события по корзине;
- **Order Service** — события по заказам;
- **Marketing Service** — рекламные кампании;
- **Promotion Service** — акции и специальные предложения;
- **Payment Service** — события по оплате;
- другие сервисы по мере роста системы.

Важно, что эти сервисы не должны напрямую отправлять PUSH.  
Их задача — публиковать доменные события.

---

### 3.6. Event Bus / Message Broker

Для передачи событий между сервисами используется брокер сообщений / шина событий, например:
- Kafka;
- RabbitMQ;
- другой аналогичный инструмент.

Через брокер передаются события вроде:
- `cart_abandoned`;
- `order_cancelled`;
- `promo_campaign_started`;
- `payment_failed`.

Это позволяет:
- развязать сервисы между собой;
- сделать обработку асинхронной;
- повысить отказоустойчивость;
- масштабировать обработчики отдельно.

---

### 3.7. Notification Service

Ключевой компонент архитектуры — отдельный **Notification Service**.

Его задачи:
- принимать события из брокера;
- определять, нужно ли отправлять уведомление;
- выбирать шаблон уведомления;
- формировать текст PUSH;
- выбирать получателей;
- учитывать настройки пользователя;
- передавать уведомление во внешний push-провайдер;
- сохранять результаты отправки.

Именно этот сервис должен быть центром всей push-логики.

---

### 3.8. Template Engine / Notification Template Storage

Для разных типов уведомлений желательно использовать шаблоны.

Примеры:
- «Вы забыли товары в корзине»
- «Ваш заказ отменен»
- «Для вас действует новая скидка»

Шаблоны могут храниться:
- в базе данных;
- в конфигурации Notification Service;
- в отдельном Template Service.

Это позволяет:
- менять тексты без переписывания бизнес-логики;
- локализовать уведомления;
- управлять версиями шаблонов.

---

### 3.9. Push Provider Integration

Для фактической доставки PUSH-уведомлений требуется интеграция с внешними push-провайдерами:
- **Firebase Cloud Messaging (FCM)** — для Android и часто как единая точка отправки;
- **Apple Push Notification service (APNs)** — для iOS напрямую, если архитектура этого требует.

Notification Service передает во внешний провайдер:
- device token;
- заголовок уведомления;
- текст;
- дополнительные данные (`payload`);
- deep link / экран для открытия.

---

### 3.10. Notification Log / Delivery Status Storage

Нужно хранить историю отправки уведомлений:
- кому отправлено;
- когда отправлено;
- какой тип уведомления;
- статус отправки;
- ошибка доставки, если есть;
- id кампании / id события.

Это важно для:
- поддержки;
- аналитики;
- повторной отправки;
- аудита;
- разбирательств по жалобам пользователей.

---
